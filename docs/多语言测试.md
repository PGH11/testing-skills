# 多语言测试的分层测试思路

## 什么是多语言测试

多语言，是指一个应用支持多种语言界面显示，让不同地区的用户能用自己的语言操作和理解内容。多语言测试是什么？就是在功能测试之外，需要确保每一种支持的语言都能正确的展示对应的翻译。

## 多语言测试的需求

1. 需要保证用户能在系统或应用内切换语言，切换语言后，界面所有文案能立即或在重新加载后切换到目标语言。

2. 所有 UI 元素（按钮、标题、菜单、提示语、报错信息等）都有正确的翻译。没有遗漏翻译（如仍显示英文 key `xxx_title`）

3. 不同语言对于同一文本的翻译，所需要的展示长度不一样。需要保证说每一种语言下都可以正常的展示，避免出现文字超出按钮范围的 UI 问题。

## 多语言测试的痛点

- 如何识别错误的翻译？ 特别是对于非母语的翻译，如果只靠人工比对是无法识别出翻译错误的文字，或者是展示的语言错误。例如当前系统语言是英式英文，而页面实际上展示的是美式英文。

- 需要覆盖的语言多。当一个系统需要覆盖上十种语言的时候，每测试一种语言都需要把所有的文本对一遍。

## 多语言测试思路分解

### 多语言的实现原理

大部分的系统中，其实多语言都只是一些不同的配置数据。

- 前端元素会绑定一个key，比如说我们产品站的页面中，“Dashboard”就绑定的是message.dashboard。

- 代码中还会为其指定一个默认值。切换别的语言时，在配置文件中找不到时，就会展示设置好得默认值。

- 每个不同的语言，都会有一个对应的 key-value 配对的配置文件（一般是 xml 或者 json），里面就存放每一个 key 对应语言的翻译内容。如英文文件中配置的是 "message.dashboard":"Dashboard", 而繁中文件中存放的是 "message.dashboard":"儀表板"

- 页面在渲染时，会根据当前用户的系统语言来请求对应语言的配置文件，并根据对应的 key 进行加载和展示。这样就实现了不同语言设置下展示对应语言翻译的效果。

总的来说，只要保证页面元素配置了正确的 key，并且所有的配置文件里的翻译内容是正确的，就基本上可以保障所有语言下的展示都是正确。

### 分层测试

根据多语言的实现原理，可以大概总结到整个迭代过程中可能出现的犯错：

- 页面元素未匹配到正确的key。 如果key 配置错误，就会导致元素和翻译内容错配的 bug。例如元素应该对应的是 “Dashboard”，但是开发错误地配置的key 是 message.chat, 就会错误地展示了 “聊天” 的翻译。

- 某个新加的元素需要增加一个新的 key，但是配置文件中没有更新对应的值。这样用户不管切换到哪个语言都只会看到默认语言的值。

- 错误地把 A 语言的翻译放到了 B 语言的的配置文件中。这样当用户设置的是简体中文时，可能会错误地看到繁体中文的翻译。

针对以上可能，可以针对性地进行分层测试：

1. 在默认语言下，根据需求文档检查每个页面的文本展示是否正确。除非某些元素没有匹配key，否则这样检查一遍可以基本保证key与需求文档是正确匹配的。

2. 切换到另一种语言。如果页面的所有文本都已经切到了不同语言，说明所有元素都通过key进行控制；如果某些元素仍然展示的是默认语言，说明元素没有正确的配置key，也就发现了bug。

3. 如果是Web端产品，这一个步骤我建议通过python爬取页面文本来做，爬取切换语言前的文本与切换语言后的文本进行对比，比对不同语言爬取的多组数组，如果相同Xpath的文本多次重复，说明该文案未配置key；如果相同Xpath文本仅重复一两次，则说明对应语言的配置文件缺少该文本的翻译。推荐使用drissionpage，一个很好用的python库

4. 如何快速测试不同的语言的配置文件正确性？
   
   一般来说配置文件及语言，通常由产品经理以表格形式提供，开发则需要把这个表格转换成不同语言的配置文件，测试的时候也可以根据这个表格进行比对。
   在流程上规范开发必须使用统一的工具进行表格 - 配置文件的转化，禁止直接复制 - 粘贴产生的配置错误。

5. 如何发现某些元素的展示错误？
   
   上面的步骤只能保证翻译配置是正确的，如果想要检查页面元素上的展示是否正确，则需要在 UI 测试的角度出发。
   在自动化用例中，在每个页面中增加一个截图的步骤，并且添加命名；在执行自动化测试时，根据语言列表，完成每种语言下都把所有的页面遍历一遍并自动截图存档。测试人员只需要快速浏览不同语言的截图，就可以找出 UI 展示的错误。
   但实际敏捷迭代中，这样成本比较大，一般来说选取一种文本最长的语言来校验各页面展示是否正常可避免绝大部分的UI问题。
