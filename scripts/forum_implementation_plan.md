# 模型失败实时监控与日志根因分析闭环方案

这份文档面向内部分享，重点讲清楚一件事：  
我们如何从“发现模型失败”走到“拿到视频 ID，再自动查询阿里云 SLS 日志并解析原因”。

整个方案不依赖人工盯页面，也不把排查动作停留在告警层，而是把监控、定位、日志检索和 AI 辅助分析串成一个闭环。

---

## 1. 背景与目标

模型生成类任务有一个典型问题：失败不是单点故障，可能来自参数、素材、模型服务、队列、三方依赖或网关超时。只看到“失败率升高”还不够，值班同学真正需要的是：

- 哪个模型或类型在失败
- 最近失败是否连续出现
- 对应的 `video_id` 是什么
- 失败时后端日志里发生了什么
- 该优先找模型服务、任务调度、素材输入还是接口链路

因此这套系统的目标不是只发一条告警，而是形成一条可追溯链路：

```text
定时拉取视频任务状态
  -> 识别异常模型和连续失败区间
  -> 提取关键 video_id
  -> 推送钉钉告警
  -> 通过 aliyun-sls-query skill 查询 SLS 日志
  -> 交给 OpenClaw 解析日志并输出排查结论
```

一句话概括：  
**用 Python 做实时监控与告警，用 `video_id` 做跨系统关联键，用 OpenClaw + SLS 查询补齐根因分析。**

---

## 2. 整体架构

当前闭环可以拆成 5 层。

### 2.1 数据采集层

通过视频任务查询接口拉取最近任务列表，核心字段包括：

- `video_id`：后续查询日志的关键索引
- `type`：模型或生成类型，用于分组统计
- `status`：任务状态，`0` 生成中、`1` 成功、`2` 失败
- `create_at`：任务创建时间，用于判断最新失败区间
- `app_type`、`device_code`：用于过滤非目标产品或外部渠道数据

设计重点：

- 分页拉取，避免接口一次返回不完整
- 先标准化字段，再进入统计逻辑
- 过滤非目标产品数据，保证告警口径稳定
- 按 `video_id` 去重，避免重复任务放大失败次数

### 2.2 异常识别层

监控逻辑不是简单统计失败数，而是优先关注“最新一段是否持续失败”。

目前核心判断有两类：

1. **连续失败阈值**：同一个 `type` 下，最新任务开始连续失败达到阈值，例如连续失败 10 次。
2. **窗口无成功判断**：最近窗口内失败数达到最低门槛，并且窗口中只有失败或生成中，没有成功记录。

这样设计的原因是：模型服务偶发失败很常见，单条失败不一定值得打扰；但“最新连续失败”更像正在发生的问题，优先级更高。

### 2.3 告警通知层

告警通过钉钉机器人发送，内容不只告诉“失败了”，而是直接给排查入口。

推荐告警结构：

```text
自动化测试 · 失败监控告警
统计范围：最近 N 条任务
需关注模型：X 个

1. 模型名称（type=xx）
   告警原因：最新连续失败达到阈值
   统计结果：连续失败 xx 次
   失败区间：oldest_time ~ newest_time
   video_id：oldest_video_id -> newest_video_id
```

这里的 `video_id` 是整个闭环里最关键的信息。它既能让业务同学定位失败样本，也能让后续日志查询有明确关键词。

### 2.4 调度运行层

使用单进程循环调度器常驻运行，负责按固定周期触发失败监控任务，同时可以保留日报任务。

当前调度策略：

- 失败监控任务：默认每 10 分钟执行一次
- 日报任务：每天固定时间执行
- 每个任务用 `safe_run` 包裹，避免单次异常导致进程退出
- 每轮执行打印开始、结束、下一次执行时间，方便定位运行状态

调度层只做编排，不写业务判断。这样后续要替换为 crontab、Supervisor、systemd 或 CI 定时任务时，核心监控逻辑不用改。

### 2.5 日志分析层

当告警出现后，系统已经给出了异常模型和关键 `video_id`。接下来通过 `aliyun-sls-query` skill 让 OpenClaw 进入日志分析流程：

1. 以 `video_id` 作为关键词查询阿里云 SLS。
2. 拉取指定时间窗口内的命中日志。
3. 解析日志中的错误栈、请求参数、服务返回、耗时和上下游调用信息。
4. 汇总成排查结论，例如：
   - 是否是素材输入异常
   - 是否是模型服务返回错误
   - 是否是任务队列超时
   - 是否是三方服务或网络异常
   - 是否是代码分支或参数兼容问题

这一步的价值在于把“告警”升级成“可操作的初步诊断”。值班同学不需要先手工打开 SLS、拼查询条件、筛日志，再猜可能原因。

## 3. 核心实现思路

### 3.1 用 `video_id` 串联业务状态和系统日志

整个方案的关键不是监控脚本本身，而是找到一个稳定的关联键。这里选择 `video_id`，因为它同时出现在：

- 视频任务接口返回结果
- 业务告警内容
- 后端服务日志
- SLS 查询关键词

只要告警里带上 `video_id`，后续所有排查动作都可以围绕它展开。

### 3.2 先判断“是否正在失败”，再判断“失败了多少”

很多监控会直接看失败率，但对生成任务来说，失败率有两个问题：

- 样本小的时候波动很大，容易误报
- 历史失败会稀释或放大当前问题，影响实时判断

所以当前策略更关注最新任务序列：

```text
按 type 分组
  -> 每组按 create_at 倒序
  -> 从最新任务开始向前扫描
  -> 遇到失败就累计
  -> 遇到生成中先跳过
  -> 遇到成功立即停止
```

这能更准确地回答一个值班问题：  
**这个模型现在是不是还在连续失败？**

### 3.3 告警只发“需要行动”的信息

内部告警最怕两种情况：

- 信息太少：只说失败，不知道怎么查
- 信息太多：一堆原始日志，没人愿意看

因此告警只保留四类信息：

- 结论：哪个模型异常
- 依据：为什么触发告警
- 范围：失败发生在哪个时间段
- 入口：用于日志查询的 `video_id`

告警的终点不是消息本身，而是下一步排查动作。

### 3.4 让 OpenClaw 做日志解析，而不是只做日志查询

`aliyun-sls-query` skill 解决的是“查得到日志”，OpenClaw 的价值是“读得懂日志”。

建议给 OpenClaw 的输入尽量结构化：

```text
请使用 aliyun-sls-query 查询以下 video_id 的日志，并分析失败原因：
- video_id: xxx
- type: xxx
- 模型名称: xxx
- 失败时间区间: yyyy-mm-dd hh:mm:ss ~ yyyy-mm-dd hh:mm:ss
- 告警原因: 最新连续失败达到阈值

请输出：
1. 命中的关键日志
2. 主要错误信息
3. 可能根因
4. 建议下一步找谁或查哪个模块
```

这样可以减少 AI 分析的不确定性，让它沿着固定排查框架输出。

## 4. 当前落地流程

### 4.1 失败监控任务

入口脚本负责拉取视频任务、分组统计、构造告警并发送钉钉。

执行流程：

1. 拉取最近视频任务数据。
2. 过滤非目标产品数据。
3. 按 `video_id` 去重。
4. 取最近固定窗口，例如最近 1000 条。
5. 按 `type` 分组。
6. 识别连续失败和窗口无成功的异常类型。
7. 构造钉钉告警文本。
8. 输出异常模型、失败区间和 `video_id`。

### 4.2 常驻调度任务

调度器负责周期性执行失败监控：

```text
启动调度器
  -> 计算下一次失败监控时间
  -> 到点执行失败监控任务
  -> 任务异常时打印错误但不中断主循环
  -> 更新下一次执行时间
```

常驻调度的好处是简单、可控、部署成本低。对于当前阶段，它比引入复杂任务平台更合适。

### 4.3 SLS 查询与分析

告警出现后，OpenClaw 根据告警里的 `video_id` 调用 `aliyun-sls-query` skill 查询 SLS。

建议查询策略：

- 优先查最新失败的 `video_id`
- 如果最新样本没有足够日志，再查失败区间内最老和最新两个 `video_id`
- 查询时间窗口覆盖失败区间前后，例如前后各扩大 10 到 30 分钟
- 解析时优先找 `error`、`exception`、`timeout`、`failed`、模型服务返回码、请求耗时等字段

输出时不要只贴日志，要给“初步判断”。例如：

```text
初步判断：失败集中在模型服务调用阶段，日志中多次出现 timeout。
建议动作：先确认模型服务可用性和队列积压情况，再检查该 type 的参数是否近期变更。
```

## 5. 关键设计取舍

### 5.1 为什么不用单条失败直接告警

生成任务天然存在偶发失败，单条失败直接告警会造成噪音。当前采用连续失败和窗口无成功，是为了让告警更接近“正在发生的问题”。

### 5.2 为什么要保留生成中状态

扫描连续失败时，生成中状态不直接打断失败序列。原因是生成中代表结果未定，如果最新队列里夹杂生成中，不能说明系统已经恢复。

### 5.3 为什么不把 SLS 查询写死进监控脚本

监控脚本负责稳定发现问题，SLS 查询负责深入排查。两者解耦有几个好处：

- 告警链路更稳定，不会被日志查询耗时拖慢
- SLS 权限、项目、日志库变化时，不影响监控主流程
- OpenClaw 可以根据不同告警动态调整查询关键词和时间窗口
- 后续可以逐步从“人工触发 OpenClaw”升级为“告警自动带上下文触发分析”

### 5.4 为什么要做字段过滤和去重

如果不先过滤 `app_type`、`device_code`，监控可能混入外部渠道或非目标产品数据。  
如果不按 `video_id` 去重，同一个任务重复出现会放大失败次数。

这两个动作本质上是保证监控口径可信。

## 6. 后续可演进方向

### 6.1 告警自动附带 SLS 查询提示词

可以在钉钉告警末尾直接生成一段可复制的 OpenClaw 指令：

```text
请使用 aliyun-sls-query 查询 video_id=xxx 的 SLS 日志，
时间范围覆盖 yyyy-mm-dd hh:mm:ss 前后 30 分钟，
并分析模型失败原因、关键错误日志和建议处理动作。
```

这样值班同学可以一键复制，降低操作成本。

### 6.2 自动生成排查摘要

下一步可以让监控脚本把告警上下文落成本地 JSON：

```json
{
  "type": 123,
  "model_name": "xxx",
  "alert_reason": "最新连续失败达到 10 次",
  "oldest_video_id": 111,
  "newest_video_id": 222,
  "oldest_time": "2026-04-30 10:00:00",
  "newest_time": "2026-04-30 10:20:00"
}
```

OpenClaw 或后续自动化任务可以读取这份上下文，再执行 SLS 查询和摘要生成。

### 6.3 引入告警冷却和升级策略

当前可以继续补充：

- 同一 `type` 在冷却期内不重复告警，避免刷屏
- 连续多个周期异常后升级优先级
- 恢复成功后发送恢复通知
- 失败持续时间过长时自动扩大 SLS 查询范围

### 6.4 建立失败原因分类

日志解析结果可以沉淀成标准分类：

- 输入素材问题
- 参数兼容问题
- 模型服务异常
- 队列积压或超时
- 第三方依赖异常
- 发布变更引入
- 未知问题，需要人工确认

后续日报可以统计“失败原因分布”，帮助研发从救火转向治理。

## 7. 内部分享建议讲法

分享时可以按下面节奏讲：

1. 先讲痛点：失败告警本身不难，难的是从告警走到根因。
2. 再讲关键思路：用 `video_id` 做业务状态和日志系统的关联键。
3. 然后讲监控策略：连续失败优先，比单点失败更适合实时告警。
4. 接着讲闭环：告警给出 `video_id`，OpenClaw 通过 `aliyun-sls-query` 查 SLS 并解析。
5. 最后讲演进：从人工复制指令，升级到自动生成查询上下文和根因摘要。

可以用这句话收尾：

> 这套方案的核心价值，不是多发一条告警，而是把“发现异常、定位样本、查询日志、形成初步判断”串成一个低成本闭环。

## 8. 风险与注意事项

- **权限风险**：SLS 查询涉及线上日志权限，skill 和 OpenClaw 的使用范围要受控，避免把敏感日志扩散到不必要的上下文。
- **误报风险**：连续失败阈值需要结合任务量调整。低流量模型阈值过高会漏报，高流量模型阈值过低会刷屏。
- **日志缺失风险**：如果服务日志没有稳定打印 `video_id`，闭环会断在 SLS 查询阶段，需要推动业务日志规范化。
- **AI 解析风险**：OpenClaw 输出应定位为“初步诊断”，关键故障仍需研发结合代码、发布记录和监控指标确认。
- **安全风险**：文档和告警中不要暴露真实 webhook、AK/SK、项目密钥和完整用户敏感数据。
